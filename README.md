# OrbitCache

Artifact for the paper **"Pushing the Limits of In-Network Caching for
Key-Value Stores"** (USENIX NSDI 2025).

OrbitCache places a small, load-balancing cache inside a Tofino
programmable switch.  Hot-key replies are generated by the switch
pipeline itself, so they never consume server CPU, and large-value
keys are handled via packet recirculation.  This repository contains
the full P4 data plane, switch-side controller, and host-side client
/ server applications needed to reproduce the paper's microbenchmarks.

## Contents

```
.
├── client.c         # open-loop client: Zipf / uniform workload generator
├── server.c         # storage-side server that owns the KV partitions
```

## Hardware dependencies

- At least two nodes (one client, one server).  More nodes are recommended so multiple clients can stress multiple emulated storage partitions.
- 100 Gbps RDMA-capable NICs supporting Nvidia VMA for kernel-bypass UDP.  Experiments still run without VMA, but throughput and tail latency will be worse.
- A programmable switch with an Intel Tofino1 ASIC.

**Tested on:**
- 8 nodes (4 clients + 4 servers), single-port Nvidia 100 GbE MCX515A-CCAT ConnectX-5 NIC.
- APS BF6064XT switch with Intel Tofino1 ASIC.

## Software dependencies

**Clients and servers:**
- Ubuntu 24.04.3 LTS with Linux kernel 6.8, 6.14, or 6.17
- Mellanox OFED NIC drivers 25.01-0.6.0 or 26.01-1.0.0
- gcc 13.3.0
- VMA libvma 9.8.40-1 or 9.8.84-1
- [TommyDS](https://www.tommyds.it/) (bundled in `tommyds/`)

**Switch:**
- Ubuntu 20.04 LTS with Linux kernel 5.4
- Python 3.8.10
- Intel P4 Studio SDE 9.7.0 and BSP 9.7.0

## Building

### Host side

1. Place `client.c`, `server.c`, `preload.c`, `header.h`, `Makefile`, and `tommyds/` in the same directory on every host (e.g., `/home/orbitcache/`).
2. Edit the cluster configuration block in `header.h` (approximately lines 36–77) so that IPs, MAC addresses, the data-plane interface name, and the controller address match your testbed.
3. Build:
   ```bash
   make
   ```
   This produces `client.out`, `server.out`, and `preload.out`.

### Switch side

1. Place `orbitcache/controller.py` and `orbitcache/orbitcache.p4` in the SDE directory on the switch control plane.
2. Edit `controller.py` (IPs, MACs, port mapping) to match your cluster.  Familiarity with Tofino / P4 is assumed.
3. Compile the P4 program against the SDE:
   ```bash
   cmake ${SDE}/p4studio \
     -DCMAKE_INSTALL_PREFIX=${SDE_INSTALL} \
     -DCMAKE_MODULE_PATH=${SDE}/cmake \
     -DP4_NAME=orbitcache \
     -DP4_PATH=${SDE}/orbitcache.p4
   make
   make install
   ```
   `${SDE}` and `${SDE_INSTALL}` are the usual Intel paths (e.g., `SDE=/home/admin/bf-sde-9.7.0`, `SDE_INSTALL=${SDE}/install`).

## Switch-side setup

Open three terminals on the switch control plane.

1. **Terminal 1 — start the P4 program.**
   ```bash
   run_switchd.sh -p orbitcache
   ```
   Tail of expected output:
   ```
   bf_switchd: dev_id 0 initialized
   bf_switchd: initialized 1 devices
   ...
   bf_switchd: server started - listening on port 9999
   bfruntime gRPC server started on 0.0.0.0:50052
   ```

2. **Terminal 2 — configure 100 Gbps ports** with `run_bfshell.sh`, then `ucli` and `pm`:
   - `port-add <port>/- 100G NONE` followed by `port-enb <port>/-` for each data-plane port.
   - Disable autoneg: `an-set -/- 2`.

3. **Terminal 3 — run the controller.**
   ```bash
   python3 controller.py
   ```
   By default the controller inserts 128 cache-lookup entries (`NUM_HOTKEY = 128`).  Tail of expected output:
   ```
   Binding with p4_name orbitcache
   Binding with p4_name orbitcache successful!!
   ...
   0 AAAAAAAAAAAAFGP 13833500700839145416 305149963468443528 0
   1 AAAAAAAAAAAAJdU  9606265380191604036 10868894308282838393 1
   ...
   127 AAAAAAAAAAAJCXe 15573036772515772950 14982745974005649426 127
   128 Keys are inserted to the lookup table.
   ```

## Host-side setup

These steps are required once per node, before any experiment.

1. **Static ARP.**  The switch does not handle host network setup, so every host that participates on the 100 Gbps data network needs a full ARP table:
   ```bash
   # Run on each host, for every peer on the data network:
   arp -s 10.0.1.101 0c:42:a1:2f:12:e6
   arp -s 10.0.1.102 0c:42:a1:2f:11:c6
   # ...
   ```
   Verify reachability with `ping` over the 100 Gbps interface before continuing.

2. **Kernel-bypass tuning** (requires root):
   ```bash
   sysctl -w net.core.rmem_max=104857600
   sysctl -w net.core.rmem_default=104857600
   echo 2000000000 > /proc/sys/kernel/shmmax
   echo 2048      > /proc/sys/vm/nr_hugepages
   ulimit -l unlimited
   ```

## Running an experiment

A complete run has four steps: start the server, seed the switch cache once with `preload.out`, start the client, and collect the log files.

### 1. Start the server

```bash
LD_PRELOAD=libvma.so VMA_THREAD_MODE=2 ./server.out <PROTOCOL_ID> [<NUM_WORKERS>]
```

| Argument        | Meaning                                                                                                      |
|-----------------|--------------------------------------------------------------------------------------------------------------|
| `PROTOCOL_ID`   | `0` = NoCache, `2` = NetCache, `3` = OrbitCache                                                              |
| `NUM_WORKERS`   | *(optional)* Emulated worker threads inside this physical server, `1..MAX_WORKERS`.  Defaults to `NUM_SRV_WORKERS` (currently `8`).  **Must match the value passed to `client.out`.** |

Example (OrbitCache, 8 emulated workers):
```bash
LD_PRELOAD=libvma.so VMA_THREAD_MODE=2 ./server.out 3
```
Tail of expected output:
```
VMA INFO: VMA_VERSION: 9.8.84-1 Release built on Jan 29 2026 23:20:06
...
Server will spawn 8 emulated worker(s)
Server 1 is running
OrbitCache is running
Tx/Rx Worker 0,Global Partition 0 is running with Socket 19
...
Data preparation done
```

### 2. Seed the switch cache

Run `preload.out` once **after** the switch controller has installed its cache-lookup entries and **before** any client traffic arrives.  It is fire-and-forget, exits in a few hundred milliseconds, and can share a terminal with a server or client binary.

```bash
./preload.out                 # NUM_HOTKEY items, default target dst_ip[0]
./preload.out <N>             # N items
./preload.out <N> <target_ip> # custom target IP
```

Expected output:
```
[preload] installing 128 items (small=64B, large=1024B, LRATIO=18%) via 10.0.1.108:3001
[preload] done: 128 items (small=104, large=24)
```

Re-run `preload.out` whenever you change `NUM_HOTKEY` / `LRATIO` or whenever the controller reinitialises the cache.

### 3. Start the client

```bash
LD_PRELOAD=libvma.so VMA_THREAD_MODE=2 \
  ./client.out <PROTOCOL_ID> <KEY_DIST> <TIME_EXP> <TARGET_QPS> <WRITE_RATIO> [<NUM_SRV_WORKERS>]
```

| Argument           | Meaning                                                                                           |
|--------------------|---------------------------------------------------------------------------------------------------|
| `PROTOCOL_ID`      | Must match the server.                                                                            |
| `KEY_DIST`         | `0` = Uniform, `1` = Zipf-0.9, `2` = Zipf-0.95, `3` = Zipf-0.99                                    |
| `TIME_EXP`         | Experiment duration in seconds (use at least 5 to absorb warm-up).                                |
| `TARGET_QPS`       | Offered rate, requests per second (goodput may differ).                                            |
| `WRITE_RATIO`      | Percentage of write requests, `0..100`.                                                           |
| `NUM_SRV_WORKERS`  | *(optional)* Must equal the value passed to `server.out`.  Defaults to `NUM_SRV_WORKERS` (`8`).   |

Example:
```bash
LD_PRELOAD=libvma.so VMA_THREAD_MODE=2 ./client.out 3 3 10 1000000 0
```
Tail of expected output:
```
VMA INFO: VMA_VERSION: 9.8.84-1 Release built on Jan 29 2026 23:20:06
...
Client using 8 emulated worker(s) per server
Client 2 is running
...
Tx Worker 0 done with 10000000 reqs, Tx throughput: 1000440 RPS
```

### 4. Collect logs

Each client writes three log files next to the binary:

| Suffix             | Content                                                |
|--------------------|--------------------------------------------------------|
| `.txt`             | Per-request latency (µs), one line per request.         |
| `-cache.txt`       | Per-request latency for switch-cache hits only.         |
| `-server.txt`      | Per-request latency for storage-backed (miss) requests. |

The last line of the primary `.txt` file holds the total wall-clock experiment time; strip it before computing latency percentiles.  See `client.c` for the exact output path convention.

## Example result

A representative OrbitCache trial from our testbed (1 physical server with 8 emulated workers, 1 client, Zipf 0.99, 10 M requests, read-only, 128 preloaded cache keys).

```
Preload              : 128 items installed (small=104, large=24)
Total requests       : 10,000,000
Experiment duration  : 9.9955 s
Throughput           : 1.00 MRPS  (= offered TARGET_QPS)
Switch-cache hits    : 3,072,290  (30.72 %)
Storage-backed hits  : 6,927,710  (69.28 %)
Dropped / unmatched  : 0
```

The ≈30 % hit rate matches the fraction of request mass concentrated on the top 128 keys of a Zipf-0.99 distribution over a 10 M-key dataset, and confirms that the switch pipeline is serving the hot set without any server-side CPU on those requests.
